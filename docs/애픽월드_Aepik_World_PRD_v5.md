## 1. 제품 개요

### 제품 목표

애니·만화·소설 / 아이돌·아티스트 / 배우·무대 / 게임 팬들이 굿즈를 쉽게 **교환**하고, **공동구매**를 모집할 수 있는 **덕질 특화 커뮤니티 플랫폼** 제공.

### 현재 문제

- 교환 상대를 트위터에서 수동으로 찾아야 함 (해시태그 검색, RT 부탁)
- 공구 모집이 SNS에 분산되어 신뢰성 확보가 어려움
- 원하는 굿즈를 찾기 어렵고 알림이 없음
- 번개장터 / 당근마켓은 "판매"에 특화되어 교환 매칭 불가
- 포카마켓은 K팝 포토카드에만 한정

### 핵심 가치

| 가치 | 설명 |
|------|------|
| **교환 매칭** | 보유/원함 기반 자동 양방향 매칭 — 경쟁사가 못 하는 것 |
| 빠른 탐색 | 키워드 알림으로 원하는 굿즈 즉시 발견 |
| 덕질 특화 | 작품/캐릭터 기반 분류로 덕후 친화적 UX |
| 신뢰 거래 | 본인인증 + 거래 평점으로 안전한 환경 |
| 멀티 팬덤 | 애니·만화·소설 + 아이돌·아티스트 + 배우·무대 + 게임 전체 포괄 |

### 왜 교환/공구부터 시작하는가?

> **전략적 판단**
> 1. **규제 최소화** — 금전 거래 중개가 없으므로 통신판매중개업 신고 부담 없음
> 2. **경쟁사 공백** — 번개장터, 포카마켓 등 모든 경쟁사가 "판매"에 집중. 교환 자동 매칭은 아무도 안 함
> 3. **커뮤니티 우선** — "돈 주고 사는 곳"보다 "덕후끼리 교환하는 곳"이 팬덤에 더 어필
> 4. **자연스러운 확장** — 교환 유저가 모이면 "그냥 팔고 싶다"는 수요가 자연 발생 → Phase 2에서 판매 추가

## 2. 1인 개발 원칙

이 PRD는 **1인 개발 환경**을 전제로 작성되었습니다.

> **1인 개발 핵심 원칙**
> - MVP는 교환 매칭 + 공구 모집 + 채팅만 — 화려함보다 동작하는 제품 우선
> - 외부 서비스 적극 활용 — 직접 개발보다 검증된 SaaS 연동
> - Phase별 엄격한 범위 관리 — 판매 기능은 Phase 2 이후
> - 기술 부채 최소화 — 나중에 혼자 유지보수 가능한 구조
> - 자동화 우선 — 반복 작업은 코드로 처리

### 직접 개발 vs 외부 서비스

| 항목 | 직접 개발 (비권장) | 외부 서비스 활용 (권장) |
|------|-------------------|----------------------|
| 본인인증 | 직접 구현 → 보안 리스크 | 아임포트 / 토스페이먼츠 API |
| 채팅 | WebSocket 직접 구현 | Firebase Realtime DB |
| 푸시 알림 | 알림 서버 직접 구현 | FCM (Firebase Cloud Messaging) |
| 이미지 저장 | 서버 직접 처리 | AWS S3 + CloudFront |

## 4. 경쟁 분석 [v5 보강]

### 시장 현황

- **K팝 굿즈 시장**: 2024년 7,913억 원 → 2025년 1조 원 돌파 (26% 성장)
- **K-POP 팬덤 시장**: 약 3조 원 규모, 5년 내 10조 원 확대 예상

### 경쟁 플랫폼 상세 분석

| 플랫폼 | 유형 | 강점 | 약점 (애픽 월드의 기회) |
|--------|------|------|-------------------|
| **번개장터** | 범용 중고거래 | MAU 100만(글로벌), 에스크로 결제, POCA Template | 수수료 6%로 인상 → 사용자 반발, 교환 기능 없음 |
| **포카마켓** | K팝 포토카드 전문 | AI 시세, 표준화 검수, 오프라인 매장 | 포토카드에만 한정, 빠른구매 10배 프리미엄, 교환 미지원 |
| **당근마켓** | 지역 중고거래 | MAU 2,000만, 편의점 배송, 압도적 점유율 | 택배 거래 약함, 덕질 카테고리·교환 기능 없음 |
| **트위터(X)** | SNS | 팬덤 커뮤니티 직접 접근 | 검색 불편, 사기 위험, 교환 관리 없음 |
| **위버스샵** | 공식 상품 판매 | 아티스트 공식 굿즈, 글로벌 5,050만 가입 | B2C 공식 판매만, 중고/교환 불가 |
| **팬플리** | K팝 커머스 | 앨범/굿즈 판매, 프로모션 | 공식 판매만, C2C 거래 없음 |

### 애픽 월드의 차별화 포지션

```
                    교환 매칭
                       ↑
                  [★ 애픽 월드 ★]
                       |
  K팝 전용 ←──────────┼──────────→ 멀티 팬덤
    (포카마켓)          |           (애픽 월드)
                       |
                    판매 중심
          (번개장터, 당근마켓)
```

**핵심 차별화:**
1. **교환 자동 매칭** — 유일한 양방향 매칭 시스템
2. **멀티 팬덤** — 애니·만화·소설 + 아이돌·아티스트 + 배우·무대 + 게임 전체 포괄
3. **수수료 없음** — 번개장터 6% 수수료 불만 유저 흡수
4. **덕질 특화 UX** — 작품/캐릭터 기반 분류, 키워드 알림

## 5. 회원가입 & 인증

### 가입 흐름 (간소화)

| 단계 | 내용 | 비고 |
|------|------|------|
| 1단계 — 소셜 로그인 | 카카오 로그인 또는 이메일 회원가입 | 로그인 없이 둘러보기 가능 |
| 2단계 — 본인인증 | 휴대폰 번호 인증 (PortOne API) | **첫 교환/공구 시도 시 요청** (가입 시 강제하지 않음) |
| 3단계 — 닉네임 설정 | 닉네임 + 프로필 이미지 설정 | 중복 불가 |
| 4단계 — 관심 카테고리 | 애니·만화·소설 / 아이돌·아티스트 / 배우·무대 / 게임 선택 (복수 선택 가능) | 선택사항, 알림 최적화용 |

> **v5 변경: Just-in-time 인증**
> - 회원가입 시에는 소셜 로그인만 필요 (가입 장벽 최소화)
> - 본인인증은 첫 교환글 등록 또는 공구 참여 시 요청
> - 비로그인 상태에서도 매물 검색·열람 가능

### 로그인 방식

| 방식 | 지원 여부 | 비고 |
|------|----------|------|
| 카카오 로그인 (OAuth) | O Phase 1 | 주 가입 방식 |
| 이메일 + 비밀번호 | O Phase 1 | 카카오 없는 사용자 대응 |
| 네이버 / 구글 / 애플 | Phase 2 | MVP 제외 |

### 회원 관리 정책

| 항목 | 내용 |
|------|------|
| 탈퇴 | 진행 중 교환/공구 완료 후 처리, 후기 보존, 개인정보 법령 기간 후 파기 |
| 휴면 | 1년 미접속 시 30일 사전 안내 후 휴면 전환, 개인정보 분리 보관 |
| 약관 동의 | 이용약관, 개인정보처리방침, 마케팅 동의(선택) |
| 미성년자 | 만 14세 미만 법정대리인 동의 필요 (PortOne에서 연령 확인) |

## 6. 신뢰 & 평점 시스템

### 거래 후기

| 항목 | 내용 |
|------|------|
| 작성 시점 | 교환/공구 완료 확정 후 7일 이내 |
| 평가 방식 | 긍정 / 부정 + 텍스트 후기 (선택) |
| 공개 여부 | 프로필에 공개 |
| 수정 불가 | 작성 후 수정/삭제 불가 |
| 이의 제기 | 부당한 후기 이의 제기 가능 (72시간 내 검토) |

### 신뢰 지표

| 지표 | 내용 | 표시 위치 |
|------|------|----------|
| 교환 횟수 | 완료된 교환/공구 건수 | 프로필, 매물 카드 |
| 긍정 비율 | (긍정 후기 / 전체 후기) × 100% | 프로필 |
| 인증 뱃지 | 본인인증 완료 표시 | 프로필, 채팅 |
| 가입일 | 계정 생성 날짜 | 프로필 |
| 최애 표시 | 최애 캐릭터/아이돌/아티스트/배우 | 프로필 |

### 신고 & 제재

| 신고 유형 | 처리 | 제재 |
|-----------|------|------|
| 교환 미발송 | 48시간 내 1차 검토 | 경고 → 7일 정지 → 영구 정지 |
| 공구 먹튀 | 48시간 내 1차 검토 | 즉시 영구 정지 + 번호 차단 |
| 허위 매물 | 48시간 내 1차 검토 | 매물 삭제 + 경고 |
| 욕설/비하 | 48시간 내 1차 검토 | 경고 → 30일 정지 |
| 사진 도용 | 48시간 내 1차 검토 | 매물 삭제 + 경고 |

## 7. 채팅 시스템

### 기술 방향

> **Firebase Realtime Database 활용**
> - 1인 개발자가 WebSocket 서버를 직접 운영하는 것은 부담이 큼
> - Firebase는 실시간 채팅 + 오프라인 지원을 무료 티어로 제공

### 채팅 기능 스펙

| 기능 | Phase 1 (MVP) | Phase 2 |
|------|-------------|---------|
| 실시간 채팅 | O | O |
| 이미지 전송 | O (S3 업로드) | O |
| 채팅 알림 | O (FCM) | O |

| 읽음 표시 | X | O |
| 채팅 차단 | X | O |
| 외부 링크 경고 | X | O |

### 교환 확정 플로우

## 8. 교환 시스템 (핵심 기능)

> **애픽 월드의 최대 차별화 기능.** 경쟁사 어디에도 없는 양방향 자동 매칭.

### 교환 유형

| 교환 유형 | 설명 | 예시 |
|-----------|------|------|
| **1:1 교환** | 내 굿즈 1개 ↔ 상대 굿즈 1개 | 카게야마 포카 ↔ 히나타 포카 |
| **다수:1 교환** | 내 굿즈 여러 개 ↔ 상대 굿즈 1개 | 일반 포카 3장 ↔ 한정판 포카 1장 |
| **1:다수 교환** | 내 굿즈 1개 ↔ 상대 굿즈 여러 개 | 피규어 1개 ↔ 포카 5장 |
| **교환 + 차액** | 시세 차이가 있을 때 금전 보충 | 카게야마 포카 + 3,000원 ↔ 히나타 포카 |

### 교환글 등록

| 필드 | 예시 | 필수 |
|------|------|------|
| 보유 굿즈 | 카게야마 포카 (하이큐 36권 부록) | O |
| 보유 수량 | 1개 ~ 여러 개 (각각 사진 첨부) | O |
| 원하는 굿즈 | 히나타 포카 동일 버전 | O |
| 상품 상태 | S급 / A급 / B급 | O |
| 사진 | 실물 사진 최대 10장 | O |
| 차액 협의 가능 | O 체크박스 (차액 협의 가능 여부) | 선택 |
| 희망 차액 | 예: +3,000원 / -5,000원 | 선택 (차액 체크 시) |
| 거래 방식 | 택배 / 직거래 / 둘 다 가능 | O |
| 직거래 위치 | 지도에서 위치 선택 + 설명 텍스트 | 직거래 선택 시 |
| 추가 조건 | 예: 상태 S급만, 동시 발송 | 선택 |

### 차액 보충 정책

> **교환 시 금전 보충이 필요한 경우**
> - 시세 차이가 있는 굿즈 간 교환 시, 차액을 금전으로 보충 가능
> - 교환글에 "차액 협의 가능" 체크 + 희망 차액 금액 입력 (선택)
> - 구체적 차액은 채팅에서 자유롭게 협의
> - 차액 정산은 이용자 간 직접 송금 (계좌/간편송금)
> - **플랫폼은 차액 대금에 개입하지 않음**

### 직거래 (오프라인 거래)

> **지도 기반 직거래 위치 표시**
> - 교환글 등록 시 거래 방식으로 "직거래" 또는 "둘 다 가능" 선택
> - 직거래 선택 시 **지도에서 대략적 위치 선택** (카카오맵 API)
> - 정확한 주소가 아닌 **역 이름, 건물 근처** 수준의 대략적 위치만 표시 (안전)
> - 추가 설명 텍스트 입력 가능 (예: "홍대입구역 2번 출구 앞", "강남역 주변 카페")
> - 매물 목록에서 직거래 가능 매물 필터링 가능
> - 지도에서 내 주변 직거래 매물 탐색 가능

### 자동 매칭 로직

```
[1:1 매칭]
사용자 A: 보유 [카게야마 포카] → 원함 [히나타 포카]
사용자 B: 보유 [히나타 포카] → 원함 [카게야마 포카]
→ 양방향 완전 매칭! → 양쪽 푸시 알림

[다수:1 매칭]
사용자 A: 보유 [카게야마 포카 3장] → 원함 [히나타 한정 포카]
사용자 B: 보유 [히나타 한정 포카] → 원함 [카게야마 포카]
→ 부분 매칭 → "교환 가능한 상대가 있어요! 세부 조건은 채팅에서 협의하세요"

[차액 매칭]
사용자 A: 보유 [일반 포카] + 차액 협의 가능 → 원함 [한정 포카]
사용자 B: 보유 [한정 포카] → 원함 [일반 포카 + 차액]
→ 매칭 알림 + "차액 협의가 필요한 교환이에요"
```

### 매칭 우선순위

1. 완전 일치 (작품 + 캐릭터 + 굿즈 타입 + 수량 모두 일치)
2. 캐릭터 일치 (굿즈 타입이나 수량은 다름 → 채팅 협의)
3. 작품 일치 (같은 작품 내 다른 캐릭터)
4. 직거래 위치 근접 시 우선순위 가산점

### 거래 진행 관리

| 단계 | 상태 | 비고 |
|------|------|------|
| 1 | 교환 합의 완료 | 채팅에서 조건 확정 (수량, 차액, 거래방식) |
| 2 | (차액 있는 경우) 차액 정산 | 직접 송금 후 입금 확인 |
| 3 | 발송 전 | 택배: 양쪽 포장 준비 / 직거래: 일정 확정 |
| 4 | 발송 완료 | 택배: 운송장 번호 입력 / 직거래: 만남 완료 |
| 5 | 수취 완료 | 상대방 확인 |
| 6 | 교환 완료 | 양쪽 확인 → 후기 작성 |

### 교환 제안 시스템

> **하나의 교환글에 여러 명이 제안을 보낼 수 있으며, 등록자가 비교 후 선택.**

#### 제안 흐름

```
[제안자]
  1. 교환 상세(EXCH02) 진입
  2. [교환 제안] 버튼 → 내 매물 선택 바텀시트
  3. 제안할 굿즈 선택 + 메시지 첨부(선택)
  4. [교환 제안 보내기] → 등록자에게 알림

[등록자]
  1. 알림 수신 or 교환 신청함(EXCH05) 확인
  2. 받은 제안 비교 (제안자 신뢰등급, 굿즈 상태, 조건)
  3. [수락] → 채팅방 자동 생성 → 교환 진행 시작
     [거절] → 사유 선택(선택) → 제안자 알림
```

#### 제안 상태

| 상태 | 설명 | 자동 전환 조건 |
|------|------|---------------|
| pending | 대기중 | - |
| accepted | 수락됨 | 등록자 수락 |
| rejected | 거절됨 | 등록자 거절 / 다른 제안 수락 시 자동 거절 |
| cancelled | 취소됨 | 제안자 취소 |
| expired | 만료 | 7일 미응답 |

#### 제안 정책

- 동일 교환글에 1인 최대 3건 제안 가능 (다른 매물로)
- 하나의 제안 수락 시 나머지 pending 제안 자동 거절
- 수락 시 양쪽 채팅방 자동 생성 + "교환 합의 진행 중" 상태 전환
- 거절 사유 예시: "다른 분과 진행합니다" / "조건이 맞지 않습니다" / "상태가 기대와 다릅니다"
- 7일 미응답 시 자동 만료 (제안자에게 알림)

#### 캐릭터 선점 설정 (진행자)

진행자는 세트에 포함된 캐릭터 목록과 각 캐릭터별 선점 가능 수량을 설정합니다.

| 캐릭터 | 선점 가능 수량 | 선점 완료 | 남은 수량 |
|--------|---------------|----------|----------|
| 히나타 쇼요 | 2 | 1 | 1 |
| 카게야마 토비오 | 2 | 2 | 0 (마감) |
| 츠키시마 케이 | 2 | 0 | 2 |
| 니시노야 유 | 2 | 1 | 1 |
| ... | ... | ... | ... |

- 세트 구매 수량(예: 2박스) → 각 캐릭터가 세트당 1개씩 포함 → 캐릭터별 선점 가능 수량 = 세트 수량
- 진행자가 캐릭터별 수량을 직접 조정 가능 (자신이 원하는 캐릭터를 제외하거나 수량 조절)
- 캐릭터 목록은 작품 DB에서 자동 로드 + 진행자가 직접 추가/삭제 가능

#### 참여자 선점 방식

- 참여자는 원하는 캐릭터를 **1개 또는 여러 개** 선택하여 선점
- 각 캐릭터별 선점 수량 지정 가능 (남은 수량 범위 내)
- 선점 시 실시간으로 남은 수량 업데이트
- 선점 취소 가능 (마감일 전까지)
- 결제 금액 = 선점한 캐릭터 수 × 캐릭터당 가격

#### 세트 공구 진행 흐름

```
[진행자: 세트 공구 등록]
  ↓ 상품명, 세트 수량, 캐릭터 목록 + 수량, 가격 설정
[참여자: 캐릭터 선점]
  ↓ 원하는 캐릭터 선택 + 수량 지정
[마감일 도달 또는 전체 선점 완료]
  ↓
[진행자: 세트 구매 진행]
  ↓
[진행자: 캐릭터별 분배 + 배송]
  ↓
[참여자: 수령 확인 + 이용후기]
```

### 공구 상태 관리 (진행자 결정권)

> **핵심 원칙:** 공구의 진행/무산 결정은 전적으로 진행자(총대)에게 있다.
> 플랫폼은 진행자의 결정을 돕는 도구를 제공하고, 참여자에게 알림을 보내는 역할만 한다.

#### 공구 상태 흐름

```
[모집중] → 마감일 도달 or 목표 달성 or 진행자 수동 마감
  ↓
[마감 — 진행 결정 대기]
  ├── 진행자 "진행 확정" → [진행중] → [배송중] → [완료]
  └── 진행자 "무산 처리" → [무산] → 참여자 전원 알림
  └── 진행자 무응답 3일 → [자동 무산 경고] → +2일 무응답 → [자동 무산]
```

#### 진행자의 능동적 결정 시나리오

| 시나리오 | 진행자 액션 | 처리 |
|---------|-----------|------|
| **목표 달성 + 진행** | "진행 확정" 버튼 | 참여자에 진행 확정 알림 + 입금 안내 |
| **목표 달성 + 무산** | "무산 처리" 버튼 + 사유 입력 (필수) | 참여자에 무산 알림 + 사유 공개, 입금자 환불 안내 |
| **목표 미달 + 진행** | "조기 마감 + 진행" 버튼 | 최소 인원 충족 시 바로 진행, 미충족 시 경고 후 진행자 확인 |
| **목표 미달 + 무산** | "무산 처리" 버튼 + 사유 입력 (필수) | 참여자에 무산 알림 + 환불 안내 |
| **모집 중 조기 마감** | "모집 조기 마감" 버튼 | 마감일 전이라도 모집을 종료하고 진행 결정 단계로 이동 |

#### 무산 처리 정책

- 무산 사유 입력은 필수 (최소 10자)
- 사유 예시: "판매처 품절", "개인 사정으로 진행 불가", "최소 인원 미달"
- 무산 시 참여자 전원에게 푸시 알림 + 인앱 알림
- 이미 입금한 참여자가 있는 경우: 무산 알림에 환불 안내 문구 자동 포함
- 무산 후 공구글은 "무산" 뱃지 표시, 목록에서 하단 이동
- 무산 횟수가 3회 이상이면 진행자 신뢰도에 영향 (프로필에 표시)

#### 진행자 잠수 방지 (자동 무산 타이머)

| 단계 | 시점 | 처리 |
|------|------|------|
| 마감일 도달 | D+0 | "진행 결정 대기" 상태 진입, 진행자에 알림 |
| 1차 리마인더 | D+1 | 진행자에 "진행/무산을 결정해주세요" 푸시 알림 |
| 2차 경고 | D+3 | 참여자에 "진행자 미응답" 알림 + 진행자에 경고 |
| 자동 무산 | D+5 | 자동 무산 처리 + 진행자에 경고 누적 (잠수 1회) |

- 잠수 2회 누적: 신규 공구 등록 30일 정지
- 잠수 3회 누적: 공구 진행자 자격 영구 박탈

### 공구 결제 방식

> **Phase 1: 플랫폼 외부 직접 송금**
> - 진행자가 참여자에게 계좌/간편송금 정보 공유
> - 플랫폼은 대금에 개입하지 않음
> - 일반 공구: 목표 미달 시 진행자가 직접 환불 처리
> - 세트 공구: 선점 현황에 따라 진행자가 구매 여부 결정
> - 무산 시 환불: 진행자가 직접 환불 처리 (플랫폼은 환불 안내 알림만 발송)

### 리스크 관리

- 진행자 자격: 교환 5회 이상 + 긍정 90% 이상
- 진행자 잠수 방지: 자동 무산 타이머 (마감일+5일) + 잠수 누적 제재
- 공지글 기능: 진행자가 참여자에게 공지사항 전달 (입금 안내, 배송 안내, 지연 안내)
- 사기 신고 시 즉시 계정 정지 + 공구 강제 종료
- 세트 공구 선점 충돌 방지: 선점 시 실시간 수량 체크 (동시 요청 시 선착순)
- 무산 남용 방지: 무산 3회 이상 시 프로필에 무산 이력 표시
- **Phase 2: 에스크로 결제 도입 검토** (토스페이먼츠)

## 11. 팬 이벤트 캘린더 [v5 신규]

> **팬들이 흩어져 있는 생일카페·지하철 광고·LED 트럭 일정을 한 곳에서 확인.**
> 트위터에 분산된 팬 이벤트 정보를 통합하여, 교환/공구를 안 하는 유저도 유입시키는 허브 역할.

### 왜 필요한가?

- 생일카페, 지하철/버스 광고, LED 트럭, 전광판 광고 등 팬 이벤트가 매주 수십 건 발생
- 현재는 트위터(X)에서 해시태그로 하나씩 찾아야 함 — 놓치는 일정이 많음
- 오프메이트·생카데이 등 기존 서비스는 K팝 전용 → 애니·만화·게임 팬 이벤트는 커버 못 함
- **애픽 월드은 멀티 팬덤 이벤트 캘린더로 차별화** (애니·만화·소설 + 아이돌·아티스트 + 배우·무대 + 게임)

### 이벤트 등록 정보

| 필드 | 필수 여부 | 비고 |
|------|----------|------|
| 이벤트 유형 | O 필수 | 생일카페 / 지하철 광고 / LED 트럭 / 전광판 / 버스 광고 / 카페 이벤트 / 기타 |
| 작품/아이돌/아티스트/배우 | O 필수 | 작품 DB에서 선택 |
| 캐릭터/멤버 | O 필수 | 대상 캐릭터 or 멤버 |
| 이벤트명 | O 필수 | 예: "카게야마 생일 카페 in 홍대" |
| 시작일 ~ 종료일 | O 필수 | 날짜 범위 |
| 운영 시간 | 선택 | 예: 11:00~20:00 |
| 장소 | O 필수 | 주소 + 지도 위치 (카카오맵 API) |
| 상세 설명 | 선택 | 특전 정보, 주의사항 등 |
| 대표 이미지 | 선택 | 이벤트 포스터 |
| 외부 링크 | 선택 | 트위터/인스타 등 공식 홍보 링크 |
| 등록자 | 자동 | 로그인 사용자 |

### 캘린더 뷰

| 뷰 | 설명 |
|-----|------|
| **월별 캘린더** | 날짜에 이벤트 도트 표시, 클릭 시 해당 날짜 이벤트 목록 |
| **리스트 뷰** | 날짜순 이벤트 목록 (오늘~30일 후) |
| **지도 뷰** | 지도에서 내 주변 이벤트 탐색 (카카오맵 API) |

### 검색 & 필터

| 필터 | 예시 |
|------|------|
| 작품/아이돌/아티스트/배우 | "하이큐", "세븐틴", "박서준" |
| 캐릭터/멤버 | "카게야마", "민규" |
| 이벤트 유형 | 생일카페, 지하철 광고 등 |
| 지역 | 서울, 부산, 오사카 등 |
| 날짜 범위 | 이번 주, 이번 달, 직접 지정 |

### 알림 연동

- 관심 작품/캐릭터의 새 이벤트 등록 시 FCM 푸시
- 키워드 알림과 통합 — 예: "카게야마" 키워드 등록 시 교환글 + 이벤트 모두 알림
- 이벤트 D-3, D-1 리마인더 (찜한 이벤트)

### 구함 기반 매칭 알림

> **구함(WANT) 게시글과 교환(EXCH) 등록글 사이의 느슨한 자동 매칭.**
> Phase 1에서는 작품+캐릭터 수준의 매칭만 구현. 버전/특전처 수준의 정밀 매칭은 Phase 2.

#### 매칭 트리거

| 트리거 | 동작 |
|--------|------|
| 교환글 등록 시 | 해당 작품+캐릭터를 "구함"으로 등록한 유저에게 알림 |
| 구함글 등록 시 | 해당 작품+캐릭터를 "보유"로 등록한 교환글 목록 즉시 노출 |
| 교환 제안 수락 시 | 구함글 등록자에게 "관련 교환이 성사되었어요" 참고 알림 |

#### 알림 메시지 템플릿

| 상황 | 메시지 | 액션 |
|------|--------|------|
| 내가 구하는 굿즈 교환글 등록 | "[작품] [캐릭터] 교환글이 등록되었어요!" | 해당 교환글로 이동 |
| 내 교환글에 매칭되는 구함글 | "[닉네임]님이 [캐릭터]를 찾고 있어요" | 해당 구함글로 이동 |
| 1:1 양방향 매칭 발견 | "완벽한 매칭! [닉네임]님과 교환 가능해요" | 교환 제안 바로가기 |

#### 매칭 조건 (Phase 1)

1. **작품 일치** (필수): 같은 작품 ID
2. **캐릭터 일치** (선택): 캐릭터까지 일치하면 우선 알림
3. **제외 조건**: 차단 유저, 이미 교환 완료된 글, 비공개 글
4. **알림 빈도 제한**: 같은 유저에게 같은 작품 알림은 24시간 1회

#### 매칭 로직 (백엔드)

```
// Cron: 새 교환글 등록 시 실행
1. 교환글의 보유 굿즈 정보 추출 (work_id, character_id)
2. WANT 테이블에서 wanted_work_id = 교환글.work_id 검색
3. character_id까지 일치하면 priority: high, 작품만 일치하면 priority: normal
4. 24시간 내 동일 알림 중복 체크
5. FCM 푸시 + 인앱 알림 발송
```

### 수익 모델 (Phase 1.5~)

| 상품 | 가격 | 설명 |
|------|------|------|
| 일반 등록 | **무료** | 기본 리스트 노출 |
| 상단 고정 | 3,000~5,000원/일 | 캘린더 최상단 고정 |
| 하이라이트 | 1,000~2,000원/일 | 색상 강조 + ★ 뱃지 |

## 12. 이벤트 배너 & 팝업 시스템 [v5 재설계]

> **팬이 직접 디자인한 이벤트(생일카페, 데뷔 기념, 콘서트 등) 홍보 배너/팝업을 사이트에 게시.**
> 배너와 팝업은 별도 상품으로 각각 신청 가능. 둘 다 신청하는 것도 가능.

### 배너와 팝업 구분

| 구분 | 이벤트 배너 | 이벤트 팝업 |
|------|-----------|-----------|
| **위치** | 홈 캐러셀 / 목록 피드 사이 | 접속 시 중앙 팝업 |
| **형태** | 가로 배너 이미지 | 세로 카드 이미지 |
| **노출 방식** | 상시 (캐러셀 자동 슬라이드) | 1일 1회 노출 (닫기 or "오늘 하루 안 보기") |
| **신청** | 별도 신청 | 별도 신청 |
| **가격** | 5,000~10,000원/일 (메인) | 3,000~7,000원/일 |

### 12-A. 이벤트 배너

#### 배너 유형

| 유형 | 크기 | 위치 | 슬롯/일 | 가격/일 |
|------|------|------|---------|--------|
| 메인 배너 (대) | 1200×300px (모바일: 375×180px) | 홈 최상단 캐러셀 | 최대 8개 | 5,000~10,000원 |
| 피드 배너 (중) | 카드 너비×180px | 교환/공구 목록 사이 (10번째 카드 위치) | 최대 4개 | 2,000~5,000원 |

#### 슬롯 시스템

- 하루에 노출 가능한 배너 수를 **슬롯**으로 제한
- 메인 캐러셀: **최대 8슬롯/일** (5초 간격 자동 슬라이드 → 40초 1사이클)
- 피드 삽입: **최대 4슬롯/일** (새로고침마다 랜덤 순서)
- 특정 날짜의 슬롯이 모두 차면 → 해당 날짜 **"매진"** 처리, 추가 신청 불가

#### 슬롯 달력 (신청 UI)

```
배너 신청 시 달력 표시:
  3/6  ████████ (8/8) — 매진
  3/7  ██████░░ (6/8) — 2자리 남음
  3/8  ████░░░░ (4/8) — 4자리 남음
  3/9  ██░░░░░░ (2/8) — 여유
→ 매진 날짜는 선택 불가 (비활성)
→ 기간 선택 시 모든 날짜에 여유 슬롯이 있어야 신청 가능
```

#### 배너 순서 (메인 캐러셀)

| 순서 규칙 | 설명 |
|----------|------|
| 1순위 | 해당일 이벤트 (오늘이 생일인 캐릭터/아이돌) |
| 2순위 | D-3 이내 이벤트 |
| 3순위 | 결제 금액 높은 순 (Phase 2 차등 가격 도입 시) |
| 4순위 | 신청 순서 (선착순) |

### 12-B. 이벤트 팝업

#### 팝업 규격

| 항목 | 내용 |
|------|------|
| 크기 | 모바일: 320×400px / 데스크톱: 400×500px |
| 위치 | 화면 중앙, 반투명 딤 배경 |
| 노출 | 접속 시 자동 (1일 1회, "오늘 하루 안 보기" 체크) |
| 슬롯 | **최대 3개/일** (여러 개면 1개씩 순차 노출, 재접속 시 다음 팝업) |
| 닫기 | [닫기] 버튼 + [오늘 하루 안 보기] 체크박스 |

#### 팝업 노출 로직

```
1. 유저 접속
2. 오늘 날짜의 활성 팝업 목록 조회 (최대 3개)
3. localStorage에서 오늘 이미 본 팝업 ID 확인
4. 안 본 팝업이 있으면 → 첫 번째 미확인 팝업 노출
5. 유저가 [닫기] → 다음 미확인 팝업 노출 (있으면)
6. [오늘 하루 안 보기] 체크 → localStorage에 날짜+팝업ID 저장 → 오늘 더 이상 안 보임
7. 모든 팝업 확인 완료 → 팝업 없음
```

#### 팝업 구성

- 이벤트 이미지 (배너보다 세로 비율, 포스터 느낌)
- 이벤트명 텍스트 오버레이 (선택)
- 하단: [이벤트 보러가기] CTA 버튼 → 이벤트 캘린더 상세로 이동
- [닫기] [오늘 하루 안 보기]

```
[팬: 배너/팝업 신청 + 결제]
  ↓
[관리자: 검수 (24시간 내)]
  ├── 승인 → 예약 게시 → 게시일에 자동 노출
  ├── 수정 요청 → 팬에게 수정 사항 안내 → 재검수
  └── 반려 → 사유 안내 + 전액 환불
[게시 기간 종료 → 자동 비노출 + 완료 알림]
```

#### 저작권 가이드라인

> **⚠️ 2026년 3월 특허청이 아이돌/배우/아티스트 무단 굿즈에 첫 시정명령을 내린 만큼 저작권 관리 필수**

| 허용 | 비허용 |
|------|--------|
| 팬아트 (직접 그린 일러스트) | 공식 사진 그대로 사용 |

| 2차 창작 일러스트 | 타인의 팬아트 무단 도용 |
| 로고 패러디 (명백한 팬 활동) | 공식 로고 그대로 사용 |

**플랫폼 보호 조치:**
- 이용약관에 "업로드 이미지의 저작권 책임은 게시자에게 있음" 명시

## 21.5. 결제 시스템 [v5 추가]

> **Phase 1에서는 결제 없이 시작. Phase 1.5부터 PortOne(구 아임포트) 기반 유료화.**

| 항목 | 내용 |

| 상품 | 가격 | 도입 시기 | 설명 |

### 끌어올리기 정책

| 항목 | 내용 |
|------|------|
| 무료 끌어올리기 | 1일 1회 (교환글, 구함글) |
| 유료 끌어올리기 | 500원/회 (1일 최대 3회) 또는 1,000원/회 (1일 무제한) |
| 동작 | 끌어올리기 시 updated_at 갱신 → 목록 최상단 이동 |
| DB | boosted_at, boost_count, boost_daily_free_used 컬럼 |
| Phase 1 | 끌어올리기 기능은 구현하되 무료 1회만 제공 (결제 없이) |

### 결제 플로우

```
[유저: 끌어올리기 / 배너 신청]
  ↓
[프론트: PortOne SDK 호출 → 결제 UI]
  ↓
[PortOne: 결제 진행 → 성공/실패 콜백]
  ↓
[백엔드: 웹훅 수신 → 결제 검증 (imp_uid)]
  ├── 성공 → 서비스 적용 (끌어올리기 실행 / 배너 예약 등록)
  └── 실패 → 에러 메시지 + 재시도 안내
```

| 상황 | 처리 |
|------|------|

| **URL 전략** | 서브패스 방식 — `/ko/exchange`, `/en/exchange`, `/ja/exchange` |
| **Phase 1** | 한국어 전용 (코드 구조만 i18n-ready로 설계) |
| **Phase 1.5** | 영어·일본어·중국어 번역 추가 |
| **번역 범위** | UI 라벨, 에러 메시지, 메타데이터(SEO), 이메일/푸시 알림 |
| **사용자 콘텐츠** | 번역하지 않음 (교환글, 공구글, 채팅 등은 원문 그대로) |
| **언어 감지** | Accept-Language 헤더 → 쿠키 저장 → 수동 변경 가능 |
| **날짜/숫자 포맷** | `Intl.DateTimeFormat`, `Intl.NumberFormat` 로케일별 자동 적용 |

> **원칙**: Phase 1부터 모든 UI 텍스트를 하드코딩하지 않고 메시지 키로 관리. Phase 1.5에서 번역 파일만 추가하면 다국어 전환 가능하도록 설계.
