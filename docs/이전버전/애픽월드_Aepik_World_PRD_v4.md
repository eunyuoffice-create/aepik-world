## 1. 제품 개요

### 제품 목표

애니 / 아이돌 / 게임 / 코스프레 팬들이 굿즈를 쉽고 빠르게 거래할 수 있는 **덕질 특화 플랫폼** 제공.

### 현재 문제

- 거래가 트위터, 에브리타임 등 SNS와 여러 플랫폼에 분산
- 원하는 굿즈를 찾기 어렵고 알림이 없음
- 교환 / 공구 관리가 불편
- 번개장터 / 당근마켓은 덕질 거래에 특화되지 않음

### 핵심 가치

| 가치 | 설명 |
|------|------|
| 빠른 탐색 | 구하기 힘든 굿즈를 키워드 알림으로 즉시 발견 |
| 덕질 특화 | 작품/캐릭터 기반 분류로 덕후 친화적 UX |
| 신뢰 거래 | 본인인증 + 거래 평점으로 안전한 거래 환경 |

## 2. 1인 개발 원칙

이 PRD는 **1인 개발 환경**을 전제로 작성되었습니다. 아래 원칙을 항상 우선합니다.

> **1인 개발 핵심 원칙**
> - MVP는 최소 기능만 — 화려함보다 동작하는 제품 우선
> - 외부 서비스 적극 활용 — 직접 개발보다 검증된 SaaS 연동
> - Phase별 엄격한 범위 관리 — 기능 추가는 Phase 2 이후
> - 기술 부채 최소화 — 나중에 혼자 유지보수 가능한 구조
> - 자동화 우선 — 반복 작업은 코드로 처리

### 직접 개발 vs 외부 서비스

| 항목 | 직접 개발 (비권장) | 외부 서비스 활용 (권장) |
|------|-------------------|----------------------|
| 본인인증 | 직접 구현 → 보안 리스크 | 아임포트 / 토스페이먼츠 API |
| 채팅 | WebSocket 직접 구현 | Firebase Realtime DB 또는 Stream Chat |
| 푸시 알림 | 알림 서버 직접 구현 | FCM (Firebase Cloud Messaging) |
| 결제 | PG 직접 연동 | 토스페이먼츠 / 카카오페이 SDK |
| 이미지 저장 | 서버 직접 처리 | AWS S3 + CloudFront |

## 4. 회원가입 & 인증

플랫폼의 신뢰성을 위해 본인인증은 필수입니다. 카카오 계정이 없는 사용자도 가입 가능합니다.

### 가입 흐름

| 단계 | 내용 | 비고 |
|------|------|------|
| 1단계 — 소셜 로그인 | 카카오 로그인 또는 이메일 회원가입 | 카카오 없어도 가입 가능 |
| 2단계 — 본인인증 (필수) | 휴대폰 번호 인증 (아임포트/토스페이먼츠 API) | 1계정 = 1번호, 미성년자 제한 가능 |
| 3단계 — 닉네임 설정 | 닉네임 + 프로필 이미지 설정 | 중복 불가 |
| 4단계 — 관심 카테고리 | 애니 / 아이돌 / 게임 / 코스프레 선택 | 알림 최적화용 |

> **본인인증 정책**
> - 1개 휴대폰 번호 = 1개 계정만 허용 (다계정 방지)
> - 인증 API: 아임포트(PortOne) 또는 토스페이먼츠 — 1인 개발자 친화적 SDK 제공
> - 미성년자(만 14세 미만) 가입 시 보호자 동의 필요
> - 번호 변경은 고객센터 통해 처리 (보안상 셀프 변경 불가)

### 로그인 방식 비교

| 방식 | 지원 여부 | 비고 |
|------|----------|------|
| 카카오 로그인 (OAuth) | O 지원 | 주 가입 방식 — 한국 사용자 친화적 |
| 이메일 + 비밀번호 | O 지원 | 카카오 없는 사용자 대응 |
| 네이버 / 구글 로그인 | Phase 2 검토 | MVP에서는 제외 |
| 애플 로그인 | Phase 2 검토 | iOS 앱 배포 시 필수 |

### [v4 보완] 회원가입 추가 검토사항

| 항목 | 내용 | 권장 Phase |
|------|------|-----------|
| 탈퇴 흐름 | 탈퇴 시 개인정보 삭제 정책, 기존 거래 기록 보존 여부 정의 필요 | Phase 1 |
| 휴면 계정 | 장기 미접속 시 휴면 전환 정책 (개인정보보호법 준수) | Phase 1 |
| 약관 동의 | 이용약관, 개인정보처리방침, 마케팅 동의 등 법적 필수 항목 | Phase 1 |
| 미성년자 가입 | 만 14세 미만 '보호자 동의'의 구체적 절차 미정의 | Phase 1 |

## 5. 신뢰 & 평점 시스템

거래 플랫폼의 핵심. 이 시스템이 없으면 사용자가 SNS로 이탈합니다.

### 거래 후기

| 항목 | 내용 |
|------|------|
| 작성 시점 | 거래 완료 확정 후 7일 이내 |
| 평가 방식 | 긍정 / 부정 + 텍스트 후기 (선택) |
| 공개 여부 | 프로필에 공개 — 누구나 확인 가능 |
| 수정 불가 | 작성 후 수정/삭제 불가 (신뢰성 유지) |

### 신뢰 지표

| 지표 | 내용 | 표시 위치 |
|------|------|----------|
| 거래 횟수 | 완료된 거래 건수 | 프로필, 매물 카드 |
| 긍정 비율 | (긍정 후기 / 전체 후기) × 100% | 프로필 |
| 인증 뱃지 | 본인인증 완료 표시 | 프로필, 채팅 |
| 가입일 | 계정 생성 날짜 | 프로필 |

### 신고 & 제재

- 사기, 허위 매물, 욕설 등 신고 기능 제공
- 신고 누적 시 관리자 검토 후 계정 정지
- MVP에서는 이메일 신고 접수 + 수동 처리 (관리자 = 나)

### [v4 보완] 신뢰 시스템 추가 검토사항

| 항목 | 내용 | 권장 Phase |
|------|------|-----------|
| 허위 후기 방지 | 실제 거래 없이 후기를 남기는 어뷰징 차단 로직 필요 | Phase 1 |
| 악의적 부정 후기 | 부당한 부정 후기에 대한 이의 제기 절차 미정의 | Phase 2 |
| 블랙리스트 | 제재받은 사용자의 재가입 방지 (번호 기반 차단) | Phase 1 |
| 신고 처리 SLA | 1인 운영 시 신고 처리 목표 시간 설정 (예: 48시간 이내) 필요 | Phase 1 |

## 6. 채팅 시스템

거래는 채팅으로 진행됩니다. 1인 개발 부담을 줄이기 위해 외부 서비스를 활용합니다.

### 기술 방향

> **추천: Firebase Realtime Database 활용**
> - 1인 개발자가 WebSocket 서버를 직접 운영하는 것은 부담이 큼
> - Firebase는 실시간 채팅 + 오프라인 지원을 무료 티어로 제공
> - 대안: Stream Chat (월 $0~, 300 MAU 무료) — 더 풍부한 채팅 UI 컴포넌트 제공

### 채팅 기능 스펙

| 기능 | Phase 1 (MVP) | Phase 2 |
|------|-------------|---------|
| 실시간 채팅 | O | O |
| 이미지 전송 | O (S3 업로드) | O |
| 채팅 알림 | O (FCM) | O |
| 거래 확정 버튼 | O (채팅 내 버튼) | O |
| 읽음 표시 | X MVP 제외 | O |
| 채팅 차단 | X MVP 제외 | O |

### 거래 확정 플로우

1. 구매자가 '거래 완료' 버튼 클릭
2. 판매자도 확인 후 '거래 완료' 클릭
3. 양쪽 확인 시 → 후기 작성 요청 알림 발송
4. 한쪽만 확인 시 → 3일 후 자동 완료 처리

### [v4 보완] 채팅 추가 검토사항

| 항목 | 내용 | 권장 Phase |
|------|------|-----------|
| 채팅 데이터 보존 | 채팅 내역 보관 기간 정의 (분쟁 대비 최소 6개월 권장) | Phase 1 |
| 외부 링크 차단 | 피싱/사기 URL 차단 또는 경고 기능 | Phase 2 |
| 계좌번호 노출 | 채팅에서 계좌번호가 평문 전송 → 보안 가이드 필요 | Phase 1 |
| 분쟁 중재 | 거래 분쟁 시 채팅 기록을 근거로 활용하는 프로세스 미정의 | Phase 2 |

## 7. 결제 & 정산 플로우

MVP에서는 플랫폼 내 결제 없이 **외부 송금**으로 진행합니다. 정산 기능은 Phase 3에서 도입 검토합니다.

### MVP 결제 방식

> **MVP: 외부 송금 방식**
> - 판매자가 채팅으로 계좌번호 / 카카오페이 / 토스 ID 공유
> - 구매자가 직접 송금
> - 입금 확인 후 판매자가 발송
> - 플랫폼은 거래에 직접 개입하지 않음 (책임 제한 명시 필요)

| 거래 타입 | 결제 방식 | 비고 |
|----------|----------|------|
| 중고 거래 | 채팅 내 계좌 공유 → 직접 송금 | MVP 기본 방식 |
| 공동구매 | 진행자가 참여자에게 계좌 공유 | 목표 미달 시 진행자가 직접 환불 |
| 교환 | 배송비만 각자 부담 | 현금 거래 없음 |

### Phase 3 — 에스크로 도입 (추후 검토)

- 토스페이먼츠 또는 카카오페이 에스크로 연동
- 구매자 결제 → 플랫폼 보관 → 배송 확인 후 판매자 정산
- 사기 분쟁 시 환불 처리 가능

### [v4 보완] 결제 추가 검토사항

| 항목 | 내용 | 권장 Phase |
|------|------|-----------|
| 책임 제한 명시 | '플랫폼은 직접 거래에 개입하지 않습니다' 법적 고지문 작성 필수 | Phase 1 |
| 사기 발생 시 대응 | 외부 송금 사기 발생 시 플랫폼 측 대응 프로세스 (신고 접수 → 증거 수집 → 계정 정지) | Phase 1 |
| 택배 운송장 관리 | 배송 추적 기능 없이 '발송 완료' 상태만 관리 시 분쟁 소지 | Phase 2 |

## 9. 작품 / 캐릭터 구조

### 등록 방식

- 작품 / 캐릭터 검색 후 선택
- 목록에 없으면 직접 추가 가능
- 별칭(alias) 지원 — 예: '내스급' → '내가 키운 S급들'

### 중복 방지 정책 (1인 개발 대응)

> **작품/캐릭터 데이터 품질 관리**
> - 누구나 추가 시 오탈자/중복 데이터 급증 위험
> - MVP: 사용자가 추가 → 관리자(나) 승인 후 공개 (Pending 상태로 임시 노출)
> - 동일 작품이 여러 이름으로 등록되면 alias 병합 처리
> - 초기 주요 작품 (하이큐, 원신, 아이브 등) 미리 시딩(Seeding)

## 12. 이미지 검색 기능 검토 [NEW]

### 기능 개요

사용자가 가지고 있는 굿즈 사진을 업로드하면, 애픽 플랫폼 내에서 동일하거나 유사한 굿즈 매물을 자동으로 찾아주는 기능입니다. 트위터나 커뮤니티에서 본 굿즈 사진으로 바로 매물을 찾을 수 있어 UX가 크게 향상됩니다.

### 사용 시나리오

- 트위터에서 본 포토카드 사진으로 즉시 매물 검색
- 보유 굿즈 사진을 찍어 교환 가능한 매물 탐색
- 굿즈 이름을 모를 때 사진만으로 검색
- 구함글 작성 시 이미지 기반 매칭

### 기술 옵션 분석

| 옵션 | 방식 | 장점 | 단점 | 비용 |
|------|------|------|------|------|
| **Google Vision API Product Search** | Google Cloud 관리형 | 높은 정확도, 모델 학습 불필요 | 굿즈 특화 아님, GCP 종속 | 1,000건/월 무료, 이후 유료 |
| **CLIP + FAISS (오픈소스)** | OpenAI CLIP + Facebook FAISS | 무료, 커스텀 가능 | 서버 GPU 필요, 세팅 복잡 | GPU 인스턴스 비용 |
| **CLIP + pgvector (PostgreSQL)** | CLIP 임베딩 + Supabase pgvector | 기존 DB 활용, 추가 서비스 불필요 | 대규모 시 성능 한계 | 기존 Supabase 비용 내 |
| **Kakao Vision API** | 카카오 관리형 | 한국 서비스, 쉬운 연동 | 범용 태깅이라 상품 매칭에 약함 | 별도 확인 필요 |

### 1인 개발 관점 종합 평가

#### 권장: Phase 2에서 `CLIP + pgvector` 방식으로 도입

**MVP(Phase 1)에서 제외하는 이유:**

1. **공수 대비 효과 불확실** — 초기에는 매물 자체가 적어 이미지 검색의 효과가 낮음. 텍스트 검색 + 키워드 알림만으로 충분한 UX 제공 가능
2. **인프라 복잡도 증가** — 이미지 임베딩 생성/저장/검색 파이프라인 구축이 1인 개발 MVP의 범위를 초과
3. **모델 서빙 비용** — CLIP 모델 서빙에 GPU 또는 CPU 인퍼런스 서버가 필요해 초기 비용 부담

**Phase 2에서 `CLIP + pgvector`를 권장하는 이유:**

1. **기존 Supabase(PostgreSQL) 인프라 활용** — pgvector 확장만 추가하면 별도 벡터 DB 없이 유사 이미지 검색 가능
2. **매물 이미지 등록 시 자동 임베딩** — 상품 등록 파이프라인에 CLIP 임베딩 생성 단계만 추가
3. **비용 효율** — FAISS 별도 서버 대비 운영 부담이 적고, 초기 매물 수만 건 규모에서는 pgvector로 충분
4. **점진적 확장 가능** — 매물이 10만 건 이상 증가하면 Pinecone이나 Qdrant 같은 전용 벡터 DB로 마이그레이션

**Phase 2 구현 시 예상 아키텍처:**

```
[사용자 이미지 업로드]
     ↓
[CLIP 모델 → 512차원 벡터 생성] (Cloud Function 또는 API 서버)
     ↓
[pgvector 코사인 유사도 검색]
     ↓
[유사 매물 목록 반환 (Top 10)]
```

**대안 — Phase 1에서 가볍게 시작하는 방법:**

자체 모델 서빙 없이 **Google Vision API**를 활용해 이미지 라벨(태그)을 자동 추출하고, 해당 태그 기반으로 텍스트 검색하는 경량 방식을 고려할 수 있습니다. '이미지 유사도 검색'은 아니지만, 사진에서 '포토카드', '아크릴', 캐릭터명 등을 자동 인식하여 검색 편의성을 높일 수 있습니다.

## 21. 배포 & 인프라 전략 [NEW]

### 배포 플랫폼 비교

| 방식 | 구성 | 장점 | 단점 | 월 비용 (초기) |
|------|------|------|------|---------------|
| **Vercel + Railway** (권장) | Vercel(Next.js) + Railway(NestJS + PostgreSQL) | Git push 자동 배포, 인프라 관리 최소, Docker 지원 | 트래픽 증가 시 비용 증가 | $0 ~ $10 |
| AWS (ECS/EC2 + RDS) | ECS(NestJS) + RDS(PostgreSQL) + S3 + CloudFront | 확장성, 가격 최적화 가능 | Docker/인프라 직접 관리, DevOps 부담 큼 | $20 ~ $50+ |
| Vercel + Supabase | Vercel(Next.js) + Supabase(DB + Auth) | 프론트+DB 거의 무료, 가장 빠른 시작 | NestJS 백엔드 별도 호스팅 필요 | $0 ~ $5 |

> **권장: Vercel(Next.js) + Railway(NestJS + PostgreSQL)**
>
> 1인 개발에서 인프라 관리는 큰 부담입니다. Railway는 Git push만으로 Docker 기반 자동 배포가 되고, PostgreSQL도 내장 제공합니다. Vercel은 Next.js에 최적화되어 있어 SSR/ISR 성능이 뛰어납니다.

### 세션 & 인증 관리

| 구성 요소 | 기술 | 역할 |
|----------|------|------|
| **PostgreSQL** | 메인 DB (Supabase 또는 Railway) | 사용자 데이터, 매물, 거래 등 영속 데이터 |
| **Redis** | Upstash (서버리스) 또는 Railway Redis | 세션 스토어, 토큰 블랙리스트, 캐시 |

> **Redis 도입 권장 이유**
>
> - JWT + Redis 조합으로 토큰 블랙리스트, 강제 로그아웃 구현 가능
> - 세션 TTL 자동 만료 지원 (인메모리라 초고속)
> - Upstash Redis는 서버리스로 무료 티어 제공 (10,000 요청/일)
> - 추후 검색 결과 캐싱, 인기 매물 캐싱에도 활용 가능
>
> **결론: PostgreSQL(영속 데이터) + Redis(세션/캐시) 이중 구조**

### 25-6. 프라모델 / 미니어처 / 도색 카테고리 확장 검토 (우선도: 낮음)

프라모델(건프라 등), 미니어처(워해머 등), 도색 커스텀 작품 등의 거래를 플랫폼에 포함하는 것을 검토했으나, **현재 단계에서는 범위에서 제외하는 것을 권장합니다.**

**제외 권장 이유:**

1. **타겟 유저층이 다름** — 애픽의 핵심 타겟은 포토카드/아크릴/피규어 등 '완성품 굿즈'를 교환·거래하는 팬덤 유저입니다. 프라모델/미니어처/도색 커뮤니티는 '조립·제작·커스텀'이 중심인 취미 문화로, 거래 패턴과 니즈가 근본적으로 다릅니다.
2. **상품 설명 복잡도 급증** — 도색 작품은 사용 도료, 작업 공정, 커스텀 부위 등 설명 필드가 크게 달라져 상품 등록 UI가 복잡해집니다. 프라모델은 스케일(1/144, 1/100, MG, PG), 미조립/조립완성/도색완성 등 별도 분류 체계가 필요합니다.
3. **가격대와 분쟁 리스크** — 커스텀 도색 작품은 수만~수십만 원대로 가격이 높고, '퀄리티 불만족'에 따른 분쟁이 잦습니다. MVP의 외부 송금 방식으로는 이런 고가 커스텀 거래의 분쟁을 처리하기 어렵습니다.
4. **1인 개발 범위 초과** — 카테고리마다 별도의 검색 필터, 태그 체계, 상태 표기 방식이 필요해 개발·운영 공수가 크게 증가합니다.

**향후 확장 가능성 (Phase 3+ 이후):**

애픽이 굿즈 거래 플랫폼으로 자리잡은 이후, 프라모델/미니어처를 '서브 카테고리'로 추가하거나 별도 탭으로 분리하는 방식은 고려할 수 있습니다. 이때는 해당 커뮤니티(건프라 갤러리, 워해머 카페 등)와 파트너십을 맺고, 전용 상품 등록 템플릿을 만드는 것이 효과적입니다.

| 시점 | 전략 |
|------|------|
| Phase 1~2 | 범위 제외 — 굿즈 거래에 집중 |
| Phase 3+ | 피규어 카테고리에 '미조립 키트' 하위 타입 추가 정도로 경량 대응 |
| 장기 | 별도 탭 또는 서브 서비스로 분리 확장 검토 |
