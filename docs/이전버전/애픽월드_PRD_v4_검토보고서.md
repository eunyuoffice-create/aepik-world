## 1. 전체 평가

PRD v4는 1인 개발 환경을 고려한 현실적인 범위 설정, Phase별 단계적 확장 계획, 기술 스택 선정 근거가 잘 정리되어 있습니다. v3 대비 법적/보안/운영 검토가 추가되어 완성도가 크게 올라갔습니다.

다만 아래 영역에서 보완이 필요합니다.

## 2. 긴급 보완 필요 사항 (Phase 1 필수)

### 2-1. 전자상거래법 개정안 반영 (우선도: 최상)

**2025년 12월 국회를 통과한 전자상거래법 개정안**에 따라, C2C 플랫폼도 통신판매중개업자로 분류되어 법적 책임이 강화되었습니다. PRD 25-1절의 "MVP는 중개가 아닌 커뮤니티형이므로 신고 불필요할 수 있으나"라는 판단은 **재검토가 필요**합니다.

| 변경 사항 | 애픽 월드에 미치는 영향 | 대응 |
|-----------|-------------------|------|
| C2C 플랫폼의 통신판매중개업자 분류 | 사업자 신고 의무 발생 가능성 높음 | 법률 자문 받아 신고 여부 확정 (Phase 1 전) |
| 구매자를 '소비자'로 규정 | 소비자보호법 적용 → 환불/교환 정책 필수 | 이용약관에 소비자 보호 조항 포함 |
| 분쟁 시 거래내역/판매자 정보 제공 의무 | 거래 로그 보존 체계 필수 | DB 설계 시 거래 이력 보존 정책 반영 |
| 과태료 상한 2배 상향 | 규정 미준수 시 리스크 증가 | Phase 1 출시 전 법적 요건 체크리스트 작성 |

**권장 액션**: Phase 1 개발 시작 전, 스타트업 전문 법률 자문을 통해 통신판매중개업 신고 의무 여부를 확정하세요. 비용이 부담된다면 중소기업청 무료 법률 상담을 활용할 수 있습니다.

### 2-2. 경쟁 환경 변화 반영

PRD의 경쟁 분석(25-7절)에 누락된 핵심 경쟁자가 있습니다.

| 플랫폼 | 현황 (2026년 기준) | 애픽 월드의 차별화 포인트 |
|--------|-------------------|---------------------|
| **포카마켓 (Pocamarket)** | K팝 포토카드 전문 플랫폼, 보유 재고 1,000만장, AI 시세 제공, 오프라인 매장(홍대 POCA SPOT) 운영 | 포카마켓은 K팝 포토카드에만 특화 → 애픽 월드은 애니/게임/코스프레까지 포괄하는 멀티 팬덤 |
| **번개장터 글로벌** | MAU 100만 돌파, POCA Template 서비스, K-WIKI 슬랭 사전 등 팬덤 기능 강화 중 | 번장은 범용 중고거래 → 덕질 특화 UX(작품/캐릭터 기반 분류)로 차별화 |
| **당근마켓** | 누적 4,000만 가입, MAU 2,000만+, 편의점 배송/직구매 기능 추가 | 지역 기반 → 전국 택배 + 교환/공구 시스템으로 차별화 |

**권장 액션**: 경쟁 분석 섹션에 포카마켓을 추가하고, 애픽 월드만의 차별점("멀티 팬덤 + 교환 매칭 + 키워드 알림")을 더 명확히 정의하세요.

## 3. 기능 누락 / 보완 항목

### 3-1. 배송 관련 기능 미정의

현재 PRD에서 배송 프로세스가 거의 정의되지 않았습니다. 굿즈 거래에서 배송은 분쟁의 주요 원인입니다.

| 추가 필요 항목 | 설명 | 권장 Phase |
|---------------|------|-----------|
| 배송 상태 관리 | 최소한 "발송 전 / 발송 완료 / 수취 완료" 3단계 | Phase 1 |
| 운송장 번호 입력 | 판매자가 운송장 번호 입력 → 구매자 확인 가능 | Phase 1 |
| 배송 추적 연동 | 스마트택배 API 등으로 자동 추적 | Phase 2 |
| 직거래 장소 가이드 | 안전한 직거래 장소 추천 (지하철역, 카페 등) | Phase 2 |
| 배송비 정책 | 배송비 포함/별도 표기 명확화 필드 | Phase 1 |

### 3-2. 신고/분쟁 처리 프로세스 구체화

5절의 신고 시스템이 "이메일 접수 + 수동 처리"로만 되어 있어, 실제 운영 시 대응이 어렵습니다.

**추가 권장 사항:**

- 인앱 신고 기능 (신고 사유 선택 + 스크린샷 첨부) — Phase 1
- 신고 접수 시 자동 알림 (접수번호 + 예상 처리 기간) — Phase 1
- 신고 유형별 처리 기준 표 (허위매물, 사기, 욕설, 도용 등) — Phase 1
- 양쪽 의견 수렴 절차 (피신고자 소명 기회) — Phase 1

### 3-3. 사용자 프로필 기능 확장

현재 프로필은 닉네임 + 이미지 + 신뢰 지표만 있습니다. 덕질 커뮤니티 특성상 아래 항목이 유용합니다.

| 추가 필드 | 설명 | 권장 Phase |
|-----------|------|-----------|
| 최애 캐릭터/아이돌 | 프로필에 "최애" 표시 → 커뮤니티 느낌 강화 | Phase 1 |
| 보유 굿즈 목록 | 교환 시 상대방이 보유 목록 확인 가능 | Phase 2 |
| 활동 뱃지 | 거래 10회, 50회, 100회 등 달성 뱃지 | Phase 2 |
| 판매 중인 매물 목록 | 프로필에서 해당 사용자의 매물 바로 확인 | Phase 1 |

### 3-4. 알림 설정 세분화

18절의 알림 시스템에서 사용자 제어 옵션이 부족합니다.

**추가 권장:**

- 알림 종류별 ON/OFF 설정 (채팅 알림은 ON, 공구 알림은 OFF 등)
- 방해금지 시간 설정 (예: 밤 11시~아침 8시)
- 알림 빈도 조절 (즉시 / 1시간 요약 / 하루 요약)

## 4. 기술 아키텍처 검토

### 4-1. Next.js 14 → 15 업그레이드 검토

PRD에 Next.js 14로 명시되어 있으나, 2026년 기준 Next.js 15가 안정화되었습니다. React Server Components, Turbopack 안정화 등의 이점이 있으므로 **시작 시점에 최신 안정 버전 사용을 권장**합니다.

### 4-2. Supabase vs Railway PostgreSQL 의사결정

24절의 미해결 과제에 DB 호스팅이 있는데, 아래 기준으로 결정을 권장합니다.

| 기준 | Supabase | Railway PostgreSQL |
|------|----------|-------------------|
| Auth 내장 | O (Supabase Auth) | X (별도 구현) |
| 실시간 구독 | O (Realtime) | X |
| 무료 티어 | 500MB, 50K MAU | $5/월 credit |
| NestJS 통합 | 약간 불편 (REST 위주) | Prisma로 직접 연결 (자연스러움) |
| 마이그레이션 | Supabase 종속적 | 표준 PostgreSQL |

**권장**: NestJS를 백엔드로 쓰는 구조에서는 **Railway PostgreSQL + Prisma**가 더 자연스럽습니다. Supabase Auth만 따로 쓰는 하이브리드도 가능하지만, 복잡도가 올라갑니다. PortOne으로 본인인증을 이미 처리하므로 Auth는 JWT 자체 구현이 깔끔합니다.

### 4-3. Firebase Realtime DB vs Supabase Realtime

채팅에 Firebase를 쓰면서 DB는 Supabase/Railway를 쓰면 기술 스택이 분산됩니다.

| 대안 | 장점 | 단점 |
|------|------|------|
| Firebase Realtime DB (현재안) | 검증된 솔루션, 오프라인 지원 | Firebase + Supabase/Railway = 이중 관리 |
| Supabase Realtime | DB 통합, 스택 단순화 | 채팅 전용으로는 기능 부족 |
| Socket.io + Redis Pub/Sub | 완전한 제어권 | 구현 공수 큼 |

**권장**: MVP에서는 Firebase가 맞습니다. 다만 Phase 2에서 채팅 고도화 시 Socket.io 전환을 검토하세요.

### 4-4. 이미지 처리 파이프라인 구체화

8절에서 S3 업로드만 언급되었으나, 실제로는 아래 파이프라인이 필요합니다.

```
[사용자 이미지 업로드]
  ↓
[서버: 파일 타입 검증 + 악성 파일 차단]
  ↓
[Sharp 라이브러리: 리사이즈(최대 1200px) + WebP 변환 + 압축]
  ↓
[S3 업로드: 원본 + 썸네일(300px) 각각 저장]
  ↓
[CloudFront CDN 배포]
```

**Phase 1 권장**: Sharp를 NestJS 서버에서 직접 실행. Lambda@Edge는 Phase 2에서 검토.

## 5. UX/비즈니스 보완 제안

### 5-1. 온보딩 흐름 개선

현재 4단계 가입 흐름이 이탈률을 높일 수 있습니다.

**개선안:**

1. 소셜 로그인 → 즉시 둘러보기 가능 (비로그인 브라우징)
2. 첫 거래 시도 시 본인인증 요청 (Just-in-time 인증)
3. 관심 카테고리는 사용 패턴 기반 자동 추천 (Phase 2)

이렇게 하면 가입 장벽이 낮아져 초기 유저 확보에 유리합니다.

### 5-2. 수익 모델 Phase 1 준비

19절에서 Phase 1 수익이 전혀 없습니다. 당장 수익화하지 않더라도 **수익화 인프라를 Phase 1에서 준비**하는 것이 좋습니다.

| 준비 항목 | 설명 |
|-----------|------|
| Google AdSense 승인 | 서비스 출시와 동시에 신청 (승인까지 2~4주) |
| 끌어올리기 DB 설계 | 매물 테이블에 boosted_at, boost_expires_at 컬럼 추가 |
| 결제 연동 테스트 | 토스페이먼츠 테스트 키로 소액 결제 흐름 검증 |

### 5-3. 커뮤니티 기능 검토

덕질 플랫폼에서 거래만 있고 커뮤니티가 없으면 트위터로 이탈합니다.

| 기능 | 설명 | 권장 Phase |
|------|------|-----------|
| 타임라인/피드 | 관심 작품의 새 매물 + 구함글이 피드 형태로 노출 | Phase 1.5 |
| 해시태그 | #하이큐 #원신 등으로 매물 탐색 | Phase 1 |
| 작품별 게시판 | 가벼운 잡담/정보 공유 공간 | Phase 3 (운영 부담 큼) |

### 5-4. SEO 전략

Next.js의 SSR/ISR 장점을 활용한 SEO 전략이 PRD에 없습니다.

**추가 권장:**

- 매물 상세 페이지 SSR + 구조화 데이터 (schema.org/Product)
- 작품/캐릭터별 랜딩 페이지 (예: /works/haikyuu → "하이큐 굿즈 거래")
- 사이트맵 자동 생성 + Google Search Console 등록
- Open Graph 메타태그 (트위터/카카오톡 공유 시 미리보기)

## 6. 보안 보완 사항

### 6-1. 계좌번호 노출 문제 (6절 v4 보완에서 언급됨)

MVP에서 채팅으로 계좌번호를 공유하는 방식은 보안 리스크가 있습니다.

**단기 대응 (Phase 1):**

- 채팅에서 계좌번호 패턴 감지 시 "계좌번호가 포함된 메시지입니다. 거래 상대방만 확인할 수 있습니다" 경고 표시
- 채팅 내역 서버 암호화 저장

**중기 대응 (Phase 2):**

- 토스/카카오페이 송금 링크 생성 기능 연동
- 계좌번호 직접 노출 없이 결제 가능한 방식 도입

### 6-2. 이미지 도용 방지

덕질 거래에서 타인의 굿즈 사진을 도용하는 사기가 흔합니다.

**권장 대응:**

- 업로드 이미지에 반투명 워터마크 자동 삽입 (닉네임 + 날짜)
- EXIF 데이터 보존 옵션 (촬영 일시 확인용)
- 동일 이미지 중복 등록 감지 (Phase 2, 해시 비교)

## 7. 운영 관련 추가 제안

### 7-1. 초기 데이터 시딩 계획 구체화

9절에서 "주요 작품 미리 시딩"이 언급되었으나, 구체적 계획이 없습니다.

**시딩 리스트 예시 (카테고리별 상위 30~50개):**

### 7-2. 베타 테스트 계획

22절의 개발 단계에 베타 테스트 계획이 없습니다.

**권장 베타 전략:**

| 단계 | 대상 | 목표 |
|------|------|------|
| 클로즈드 베타 (2~4주) | 트위터 덕질 계정 팔로워 중 50~100명 모집 | 핵심 기능 버그 탐지, UX 피드백 |

| 정식 출시 | 전체 공개 | 마케팅 캠페인 시작 |

### 7-3. KPI 정의

25-5절에서 DAU, 거래 건수 등이 언급되었으나 목표치가 없습니다.

## 8. DB 스키마 설계 권장

PRD에 DB 스키마가 정의되어 있지 않습니다. Phase 1 착수 전에 핵심 테이블을 먼저 설계하는 것을 권장합니다.

**핵심 테이블 목록:**

- `users` — 회원 정보, 인증 상태
- `products` — 매물 (제목, 설명, 가격, 상태, 거래방식)
- `product_images` — 매물 이미지 (S3 URL, 순서)
- `works` — 작품 (이름, 별칭, 카테고리, 승인상태)
- `characters` — 캐릭터 (이름, 별칭, 작품 FK)
- `goods_types` — 굿즈 타입
- `chats` / `chat_messages` — 채팅방, 메시지
- `transactions` — 거래 (상태, 구매자, 판매자)
- `reviews` — 후기
- `reports` — 신고
- `keyword_alerts` — 키워드 알림
- `wishlist` — 찜
- `exchanges` — 교환
- `group_buys` — 공동구매
